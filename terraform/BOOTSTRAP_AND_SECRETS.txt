================================================================================
GitHub Actions — repository secrets (Settings → Secrets and variables → Actions)
================================================================================

Rubric (AWS credentials):
  AWS_ACCESS_KEY_ID       IAM user or CI user access key
  AWS_SECRET_ACCESS_KEY   Matching secret key
  AWS_SESSION_TOKEN       Required by rubric. Use the token from STS/assumed-role
                          output. For long-lived IAM users with no session token,
                          create a secret whose value is a single space or ask
                          your instructor whether the field may be omitted in
                          configure-aws-credentials.
  AWS_REGION              e.g. us-east-1

Terraform remote state (required before the main workflow can apply):
  TF_STATE_BUCKET         S3 bucket name from terraform/bootstrap (one-time apply)
  TF_STATE_KEY            Object key for state file, e.g. devops-ecom/terraform.tfstate
                          If omitted, the workflow defaults KEY to devops-ecom/terraform.tfstate.
  TF_STATE_DYNAMODB_TABLE DynamoDB table name for state locking (from bootstrap)

  TF_STATE_REGION         Optional. Region of the STATE bucket and lock table ONLY.
                          If unset, the workflow uses AWS_REGION for terraform init.
                          Set this when your bucket lives in a different region than your app
                          (fixes S3 ListObjectsV2 StatusCode 301: "actual location us-east-1").

  Example mismatch: bootstrap created the bucket in us-east-1, then AWS_REGION was changed
  to ap-south-1 for ECS — terraform init must still use region=us-east-1 for the backend.
  Fix: set TF_STATE_REGION=us-east-1 (or recreate bootstrap + bucket in ap-south-1 and use one region everywhere).

Application secrets (AWS Secrets Manager — create secrets in console/CLI first).
The workflow maps these GitHub Actions secret names to Terraform TF_VAR_*:

  DATABASE_URL_SECRET_ARN          Full ARN of secret holding DATABASE_URL (MongoDB URI)
  JWT_SECRET_ARN                   Full ARN of secret holding JWT_SECRET
  CLOUDINARY_CLOUD_NAME_SECRET_ARN Optional
  CLOUDINARY_API_KEY_SECRET_ARN    Optional
  CLOUDINARY_API_SECRET_SECRET_ARN Optional

Use GitHub "Environments" or masked secrets; never commit ARNs with plaintext values.

ECS: Create each secret in Secrets Manager (plain string: paste the URI or JWT value). Copy the
secret's full ARN into GitHub. After the next full pipeline run, Terraform grants the ECS task
execution role GetSecretValue on those ARNs, and the deploy job injects them into the task
definition. Without DATABASE_URL, the Node app exits on startup and ECS tasks loop as "Essential
container exited".

Atlas: allow network access from Fargate (for coursework often IP Access List 0.0.0.0/0).

================================================================================
One-time: Terraform remote state bootstrap
================================================================================

From terraform/bootstrap/ (local backend is fine for this tiny stack):

  cd terraform/bootstrap
  terraform init
  terraform apply -var="region=ap-south-1"

Copy outputs tfstate_bucket_name and tfstate_lock_table into GitHub secrets
TF_STATE_BUCKET and TF_STATE_DYNAMODB_TABLE. Set TF_STATE_KEY to a stable path.

--------------------------------------------------------------------------------
Option B — New bootstrap only in ap-south-1 (Mumbai), one region everywhere
--------------------------------------------------------------------------------

1) AWS CLI / credentials: use an identity allowed to create S3 + DynamoDB in ap-south-1.

2) If you still have terraform/bootstrap local state for the OLD region (e.g. us-east-1):
     cd terraform/bootstrap
     terraform destroy -var="region=us-east-1"
   Use the region where that bucket was ORIGINALLY created (see S3 console "AWS Region" for the
   bucket). If your variables.tf default is ap-south-1 but the bucket is in us-east-1, destroy
   WITHOUT -var fails with S3 StatusCode 301 / PermanentRedirect.
   If destroy fails (bucket not empty): empty the state bucket in the console, then destroy again,
   or delete the old bucket + DynamoDB table manually in the OLD region.

3) Start a fresh bootstrap state in Mumbai (pick ONE):
   A) Remove local state then re-init:
        cd terraform/bootstrap
        rm -f terraform.tfstate terraform.tfstate.backup .terraform/terraform.tfstate
        terraform init
        terraform apply -var="region=ap-south-1"
   B) Or keep default region in terraform/bootstrap/variables.tf as ap-south-1 and:
        terraform apply

4) Copy new outputs into GitHub:
     TF_STATE_BUCKET        <- tfstate_bucket_name
     TF_STATE_DYNAMODB_TABLE <- tfstate_lock_table_name
     AWS_REGION             <- ap-south-1
   Remove secret TF_STATE_REGION if you added it earlier (so init uses AWS_REGION only).

5) Main stack (terraform/): first CI run uses an EMPTY state in the new bucket — it will
   create ECR/ECS/ALB in ap-south-1. If you already applied the main stack in another region,
   destroy that old stack first (old backend + terraform destroy) or you will pay for duplicate
   resources and possible name collisions.

================================================================================
MongoDB (Atlas) and ECS
================================================================================

  - Use MongoDB Atlas (or compatible) and allow inbound from AWS: either "0.0.0.0/0"
    for a class project (not production) or VPC peering / PrivateLink for stricter
    setups.
  - Store DATABASE_URL in Secrets Manager; grant the ECS task execution role read
    access (handled by main Terraform when ARNs are set in TF_VAR_* secrets).
